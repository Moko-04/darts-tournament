# 店舗サポート 報告書アプリ 要件定義

最終更新: 2026-06-16 ／ 対象: `store-support-report`（単一HTMLアプリ）

---

## 1. 目的・背景
店舗サポート事業部の清掃等の作業報告書を、**現場の携帯だけで完結**して作成・送付できるようにする。
従来は写真整理とPDF作成（例: 「3月茅場町.pdf」）に手間がかかっていた。これをアプリ化し、
**入力 → PDF自動生成 → LINEで共有**までをワンストップにする。

## 2. 対象ユーザー・利用環境
- **利用者**: 店舗サポート事業部の作業担当者（現場）。
- **端末**: スマートフォン（iPhone/Android）主体。ホーム画面追加で簡易アプリ運用。
- **回線**: モバイル回線。CDN/Supabase利用のためオンライン前提（入力途中はローカル保持）。

## 3. スコープ
### 対象（やること）
- 報告書の作成（店舗・日時・担当・作業項目・写真・特記事項）
- 施工前/施工後 写真の添付（カメラ撮影 or ファイル選択）
- PDF自動生成、LINE送信、報告書の保存（履歴）
- マスタ管理（担当者・店舗・項目カテゴリ・並び順・店舗別の定番項目）
### 対象外（今回やらないこと）
- 請求/見積など会計処理、勤怠、顧客管理
- 報告書の承認ワークフロー（上長承認など）
- 多言語対応

## 4. 機能要件

### 4.1 報告書作成フロー（2ステップ）
- **ステップ① 基本情報**: 店舗（選択）／日時（既定=現在、変更可）／作業担当者（複数選択可）。
- **ステップ② 作業項目**: 定番項目の一覧から選んで写真入力。上部に①の内容サマリ（「修正」で①へ戻れる）。
- 進捗をステップバー（① ✓ → ②）で表示。

### 4.2 作業項目（タップ式チェックリスト）
- あらかじめ決まった**項目（カテゴリ）一覧**を表示し、**項目名タップで写真入力画面**に入る。
- 一覧の各行に状態を表示: **完了=緑✓＋枚数 / 写真不足=橙！ / 未入力=灰＋**。
- 一覧に無い単発作業は「**その他の項目を追加**」で名称入力して追加可能。
- 編集画面に「**次の項目へ →**」を備え、連続入力できる（最後の項目では「一覧へ」）。

### 4.3 写真
- 各項目に **施工前 / 施工後** の枠。各枠で「**📷カメラ**（その場撮影）」「**🖼ファイル**（端末から選択・複数可）」を選べる。
- 端末側で自動縮小（長辺1400px・JPEG）して容量を抑える。サムネ表示＋個別削除。
- **必須要件**: 各使用項目は**施工前・施工後の両方**の写真が必要。揃わない項目があると「完了」できず警告。

### 4.4 完了処理（「完了して送信」）
1. 入力内容から**PDFを自動生成**（ヘッダー表＋作業ごとに施工前/後を並べる。例PDFと同等レイアウト）。
2. （クラウド設定時）写真の表紙画像とPDFを**Supabase Storageへアップロード**。
3. **LINE Messaging API** で送信先（グループ/ユーザー）へ push（本文＋PDFリンク＋表紙画像）。
4. 報告書を**履歴に保存**。
5. クラウド/LINE未設定時は、端末の**共有シート（Web Share）またはダウンロード**にフォールバック（オフラインでも完結）。

### 4.5 マスタ管理（設定タブ）
- **担当者**: 追加・削除。
- **店舗**: 追加・削除。
- **項目カテゴリ**: 追加・削除＋**▲▼で並び替え**（作成画面はこの順で表示）。
- **店舗ごとの定番項目**: 店舗を選び、よくやる項目をオン。作成時にその店舗では上部へ「★この店舗の定番」としてまとめて表示。

### 4.6 履歴
- 作成済み報告書を新しい順に一覧（店舗・日時・担当・項目）。
- **PDF再共有**、（クラウド保存時）**クラウドPDFリンク**、**削除**。

### 4.7 認証・クラウド保存（任意・Supabase設定時）
- ログイン必須（管理者がアカウントを手動発行。自己登録なし）。
- マスタ・報告書をユーザー単位でクラウド同期（RLSで本人のみ）。
- 未設定時はログイン無し・端末（localStorage）保存のみで全機能動作。

## 5. 非機能要件
- **モバイル最適化**: 390px幅基準、下部タブ＋固定の主要ボタン。スクロールが伸びない設計（項目は折りたたみ/一覧式）。
- **オフライン耐性**: 入力～PDF生成～共有は端末内で完結可能。クラウド/LINEは設定時のみ。
- **ビルド不要**: 単一HTML＋CDN。`file://` 直開きやGitHub Pagesで配信可能。
- **セキュリティ**: クライアントに置くのは公開キー（Supabase publishable）のみ。LINEトークン等の秘密はサーバ側（Edge Functionのシークレット）に保持。リポジトリに秘密情報を含めない。

## 6. データモデル（概要）
- **マスタ**: `staff[]`（担当者）／`stores[]`（店舗）／`categories[]`（項目・順序保持）／`storeDefaults{ 店舗: [カテゴリ] }`（店舗別定番）。
- **報告書 `report`**: `{ id, store, date, staff[], items[], note, createdAt, pdfUrl? }`
  - `items[]`: `{ category, before[], after[], note }`（before/afterは写真。端末では縮小JPEGのdataURL、クラウド時は公開URL）。

## 7. 外部連携
- **Supabase**: 認証（メール/パスワード）、`app_state`（マスタ）、`reports`（任意）、Storage `reports`（写真/PDF）。RLSで本人のみ。
- **LINE Messaging API**: 公式アカウント＋チャネルアクセストークン。Edge Function `send-line` がサーバ側で push。送信先 `LINE_TO`（グループ/ユーザーID）。

## 8. システム構成・技術スタック
- フロント: **単一 `index.html`**。React 18 + Babel standalone + Tailwind（CDN）。
- PDF生成: **html2canvas + jsPDF**（DOM→画像→PDF。日本語対応）。
- バックエンド（任意）: **Supabase**（Auth / Postgres / Storage / Edge Functions）。
- 配信: GitHub Pages（`.nojekyll`、HTTPS）。

## 9. 制約・前提
- CDN利用のためオンライン必須（初回ロード）。
- LINEはメッセージでPDFを直接添付できないため、**PDFはリンク**＋表紙画像で送付。
- リポジトリは **`store-support-report`（Public）** に独立させる方針（GitHub Pagesで配信）。

## 10. 今後の拡張候補
- LINE送信先を店舗ごとに切替（店舗別グループ）。
- 履歴のクラウド一覧表示・検索。
- 報告書テンプレ（会社ロゴ・捺印欄）、3位…等の体裁調整。
- アプリ内パスワード変更、権限（管理者/一般）。

## 11. 現状ステータス（2026-06-16）
- 実装済み: 2ステップ作成 / タップ式チェックリスト / カメラ・ファイル選択 / 写真必須化 / 次の項目へ / 並び順 / 店舗別定番 / PDF生成 / 履歴 / マスタ / ログイン雛形 / LINE Edge Function 雛形。実機相当（390px）で動作確認済み。
- 未設定（運用時に必要）: Supabaseプロジェクト接続、LINEチャネル＋Edge Functionデプロイ、独立リポジトリへの配置とPages公開。
